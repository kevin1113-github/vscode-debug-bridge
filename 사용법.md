# VSCode Debug Bridge — 한국어 사용법

Claude Code / Codex 같은 AI 에이전트가 **VSCode 디버거를 직접 조작**할 수 있게 해주는 브리지입니다.
브레이크포인트 설정 → 정지 → 콜스택·변수 검사 → 표현식 평가 → 스텝/재개까지 전부 도구 호출로 가능합니다.

Unity(`vstuc`)뿐 아니라 Python·Node·C++ 등 모든 DAP 디버거에 동작하지만, 이 문서는 **Unity C# 디버깅** 기준으로 설명합니다.

```text
에이전트 ──MCP──> mcp/server.js ──HTTP(127.0.0.1:39517)──> VSCode 확장 ──vscode.debug──> Unity(vstuc)
```

---

## ⚠️ 먼저 알아야 할 개념 (혼동 주의)

### 1. "Attach"는 수동적 연결이다 — 붙어도 아무 일도 안 일어난다
디버거를 attach하면 그냥 **연결만** 됩니다. 게임이 자동으로 실행(Play)되지도, 화면이 멈추지도 않습니다.
연결이 잘 됐는지는 VSCode 하단 상태바가 **주황색**으로 바뀌고 디버그 툴바가 뜨는 것으로 확인합니다.

### 2. Unity가 "얼어붙는" 건 오직 브레이크포인트에 걸리는 순간뿐
브레이크포인트는 **실제로 실행 중인 코드**가 그 줄을 지나야 걸립니다.

- **Edit 모드**(Play 안 누름) → 게임플레이 코드가 안 돌아감 → 브레이크포인트 안 걸림 → "가만히 있는" 게 정상
- **Play 모드** → 게임 로직이 그 코드를 호출하는 순간 → 걸림 → Unity 메인스레드 정지(에디터 "응답 없음")

### 3. "일시정지"는 두 종류 — 헷갈리지 말 것

| 구분 | 정체 | 증상 | 푸는 법 |
|------|------|------|---------|
| **Play 일시정지** | Unity의 Pause 버튼 / `EditorApplication.isPaused` | 에디터는 멀쩡, 게임만 프레임 정지 | Pause 버튼 끄기 |
| **디버거 정지** | 브레이크포인트에 걸림 | 에디터 전체가 "응답 없음", VSCode에 노란 화살표 | `debug_continue` |

> 🔴 **중요한 함정**: Play 모드에서 브레이크포인트로 멈춘 뒤 `continue`/`detach`만으로는
> Unity의 일시정지 플래그가 안 풀려 게임이 멈춘 채 남는 경우가 있습니다.
> → 디버깅 후 게임이 멈춰 있으면 **Unity Pause 버튼을 끄거나** `EditorApplication.isPaused = false`로 풀어주세요.

### 4. 정지 중엔 UnityMCP가 "연결 불가"로 나온다 — 정상
브레이크포인트에 걸리면 Unity 메인스레드가 얼어서 UnityMCP가 응답 못 합니다.
이때 검사는 `debug_*` 도구(별도 채널)로 하고, **재개한 뒤에야** UnityMCP를 쓰세요.
에디터가 "응답 없음"이라고 강제 종료하지 마세요 — 그냥 브레이크포인트에 걸린 겁니다.

## 도구 목록

| 도구 | 기능 |
|------|------|
| `debug_status` | 세션 상태 (연결됨/정지됨/스레드/브레이크포인트 수) |
| `debug_attach` | launch 설정 이름으로 어태치 (예: `"Attach to Unity"`) |
| `debug_detach` | 세션 종료 |
| `debug_set_breakpoint` | 브레이크포인트 설정 (절대경로 + 1-base 줄번호, 조건식 옵션) |
| `debug_list_breakpoints` | 브레이크포인트 목록 |
| `debug_clear_breakpoints` | 특정 파일 또는 전체 브레이크포인트 제거 |
| `debug_continue` | 재개 |
| `debug_step_over` | 한 줄 실행 (DAP `next`) |
| `debug_step_in` | 함수 안으로 진입 |
| `debug_step_out` | 함수 밖으로 |
| `debug_pause` | 실행 중 일시정지 |
| `debug_stack` | 콜스택 (정지 상태에서만 의미 있음) |
| `debug_variables` | 프레임의 지역변수/스코프 (기본 최상위 프레임) |
| `debug_expand` | 구조체/객체 변수 펼치기 (variablesReference) |
| `debug_evaluate` | 표현식 평가 (프레임 컨텍스트) |

## 표준 작업 흐름 (Unity 게임플레이 디버깅)

### 흐름 A — 실행 중인 게임에서 자연스럽게 잡기 (권장)

```
1. (Unity에서) Play 모드 진입       ← UnityMCP manage_editor(play) 또는 수동
2. debug_attach { config: "Attach to Unity" }
3. debug_set_breakpoint { file: "<절대경로>.cs", line: N }
4. 게임이 그 코드 경로를 실행 → 자동으로 정지
5. debug_status            → stopped:true 확인
6. debug_stack             → 어디서 멈췄는지
7. debug_variables / debug_evaluate  → 값 검사
8. debug_step_over / step_in / continue → 실행 제어
9. (정리) debug_clear_breakpoints → debug_continue → debug_detach
10. 게임이 멈춰있으면 isPaused=false 로 해제
```

### 흐름 B — 특정 코드를 확실히 걸리게 만들기 (결정적 트리거)

자동으로 안 걸리거나 특정 함수만 콕 집어 멈추고 싶을 때, `execute_code`로 호출을 예약합니다.
**핵심**: 직접 호출하면 메인스레드가 얼어 호출이 블록되므로, `delayCall`로 예약만 하고 즉시 리턴시킵니다.

```csharp
// UnityMCP execute_code 로 실행
UnityEditor.EditorApplication.delayCall += () => {
    EmberKnight.Battle.Tests.BattleSelfTest.Run();  // 브레이크포인트 있는 코드 호출
};
return "scheduled";
```
→ 다음 에디터 틱에 실행되며 브레이크포인트에 걸림. 이후 `debug_status`로 정지 확인 후 검사.

## 자주 겪는 문제 (FAQ)

**Q. attach 했는데 아무 변화가 없어요.**
A. 정상입니다. attach는 연결만 합니다. 브레이크포인트를 걸고 그 코드가 실행돼야 멈춥니다.

**Q. 게임 화면이 안 움직여요.**
A. ① Play 모드가 아니거나(Edit 모드), ② Play인데 일시정지(Pause) 상태입니다.
   디버거 문제가 아닙니다. `isPlaying` / `isPaused`를 확인하세요.

**Q. 브레이크포인트가 안 걸려요.**
A. ① Play 모드인지, ② 그 코드 경로가 실제로 실행되는지, ③ 줄번호가 실행 가능한 코드 줄인지 확인.
   주석/빈 줄/선언부엔 안 걸립니다.

**Q. 디버깅 끝났는데 게임이 계속 멈춰 있어요.**
A. Play 모드 브레이크포인트의 잔류 일시정지입니다. Unity Pause 버튼을 끄거나
   `EditorApplication.isPaused = false`로 푸세요.

**Q. UnityMCP가 "No Unity Editor instances / Could not connect" 에러를 내요.**
A. 브레이크포인트에 걸려 메인스레드가 멈춘 상태일 가능성이 큽니다.
   `debug_status`로 stopped 확인 → `debug_continue`로 재개하면 UnityMCP가 다시 응답합니다.

## 멀티 프로젝트 / 포트 (★옵션 A — 프로젝트별 포트)

브리지 HTTP 서버는 **단일 포트**를 씁니다. VSCode 창은 각자 확장 호스트가 따로 돌아
각 창이 같은 포트를 잡으려 하므로, **먼저 포트를 잡은 창 하나만** 제어됩니다.
여러 창을 동시에 디버깅하려면 **프로젝트마다 포트를 다르게** 주고, **MCP도 프로젝트 스코프로** 등록합니다.

### 프로젝트 1개 세팅 레시피

1. **워크스페이스 포트 지정** — 해당 프로젝트의 `.vscode/settings.json`:
   ```json
   { "debugBridge.port": 39517 }
   ```
2. **프로젝트 스코프 MCP 등록**(전역 user-scope는 쓰지 않음) — 프로젝트 폴더에서:
   ```bash
   claude mcp add --scope project vscode-debug-bridge \
     --env DEBUG_BRIDGE_PORT=39517 \
     -- npx -y github:kevin1113-github/vscode-debug-bridge
   ```
   → 프로젝트 루트에 `.mcp.json` 생성. **포트(`DEBUG_BRIDGE_PORT`)는 1번의 `debugBridge.port`와 반드시 일치.**
3. Claude Code가 프로젝트 `.mcp.json` 서버를 처음 쓸 때 **"승인 대기"** → 승인하면 사용.

### 포트 배정 표 (충돌 방지 — 프로젝트마다 고유 포트)

| 프로젝트 | `debugBridge.port` = `DEBUG_BRIDGE_PORT` |
|----------|------------------------------------------|
| TheEmberKnight | `39517` |
| (프로젝트 2) | `39518` |
| (프로젝트 3) | `39519` |

> 전역(user-scope) 등록은 단일 포트로 고정돼 멀티 창과 충돌하므로, 옵션 A에서는 쓰지 않습니다.
> 한 번에 한 프로젝트만 디버깅한다면 굳이 옵션 A가 필요 없습니다(기본 단일 포트로 충분).

## 설치 / 등록

설치·등록 방법은 [`README.md`](README.md) 참고. 요약:
- 확장: `vsce package` → `code --install-extension *.vsix` (한 번만, 전역)
- MCP: `claude mcp add --scope user vscode-debug-bridge -- node <경로>/mcp/server.js` (단일 창)
  - 멀티 창 동시 디버깅이면 user-scope 대신 **위 "멀티 프로젝트 / 포트" 옵션 A**(프로젝트별 포트 + `--scope project`)
- Codex: `~/.codex/config.toml`의 `[mcp_servers.vscode-debug-bridge]`에 등록
